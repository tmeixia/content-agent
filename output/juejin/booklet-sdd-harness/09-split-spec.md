# 立项与拆 Spec：把简版小红书拆成可交付单元

上一篇我们用 5 分钟跑通了第一个 `Spec→代码`（#10）。从这篇开始，我们正式动手做贯穿全册的实战项目——**简版小红书（代号 RedLite）**，只做四个功能：登录注册、发帖表单、帖子列表、帖子详情。

先抛个结论，这篇就讲透它：

> **拿到需求别急着让 AI 一把梭。第一步永远是——把它拆成对的 Spec 单元。拆错了，AI 输出质量断崖下降。**

## 为什么不能"一份大 Spec 丢给 AI"

我见过太多人（包括早期的我）：拿到需求，写一份三千字的大 Spec，把所有功能塞进去，丢给 AI，期待它一把生成整个项目。

结果是什么？所有组件挤在一个文件里、状态互相缠绕、改一处全崩。解耦花的时间比重写还长。

原因很简单：**AI 的注意力和你一样有限。** 一次让它扛太多，它就开始顾此失彼。Spec 的粒度，直接决定 AI 的转化率。

## 怎么拆才对：三条标准

判断一个功能该不该独立成一个 Spec 单元，过三条标准：

1. **独立可测试** —— 这块做完，能单独跑验收吗？
2. **数据边界清** —— 它消费的数据来自同一个源吗？
3. **可独立交付** —— 做完，页面上能看到效果吗？

> 打个比方：**乐高说明书每一步只拼一个子件，绝不会把整辆车画在一页上。** 拆 Spec 就是给 AI 画"每次只拼一块"的说明书。

## 可抄走的硬货 ①：让 AI 帮你拆的完整 Prompt

注意——不是甩一句"帮我拆一下"，而是**把拆分标准和输出格式一起给它**。你越具体，AI 越不会乱来。直接抄：

```
我正在开发一个简版图文社区 App（类小红书），MVP 包含以下功能：

1. 登录注册：手机号/邮箱注册、登录、JWT 鉴权
2. 发帖表单：标题 + 正文 + 封面图上传，提交后进入我的帖子
3. 帖子列表：信息流卡片（封面+标题+作者+时间），支持分页
4. 帖子详情：点击卡片进入详情页，展示完整正文与封面

请将它拆分为独立的 Spec 开发单元。
拆分标准：每个单元可独立开发、独立测试、独立交付。
输出格式（表格）：
- 单元名（specs/redlite-xxx.md）
- 一句话描述
- 数据依赖（静态 / 需要 API）
- 与其他单元的依赖关系（有 / 无，依赖谁）
```

## AI 输出 + 讲评

AI 给出 5 个单元（示意）：

| 单元 | 描述 | 数据依赖 | 依赖关系 |
|---|---|---|---|
| `redlite-auth` | 登录注册 + JWT | 需要 API | 无（最先做） |
| `redlite-post-create` | 发帖表单 + 图片上传 | 需要 API | 依赖 auth（要登录态） |
| `redlite-post-list` | 信息流列表 + 分页 | 需要 API | 无（可并行） |
| `redlite-post-detail` | 帖子详情页 | 需要 API | 依赖 list（从卡片进入） |
| `redlite-post-card` | 列表/详情共用的卡片组件 | 静态 | 被 list/detail 复用 |

**讲评（人来做判断）**：
- ✅ 把"卡片组件"单独抽出来很对——列表和详情都要用，抽出来避免重复。
- ✅ auth 标了"最先做"，没毛病，发帖要登录态。
- ⚠️ 如果 AI 把"图片上传"拆成独立单元，我会**合并回 post-create**——上传只是发帖表单的一个字段，没必要独立。

> **结论：方案基本接受。** 要调整，追加一句"把 X 和 Y 合并/拆开"就行，不用重来。

## 可抄走的硬货 ②：生成单个 Spec 的 Prompt

拆分方案定了，让 AI 为第一个单元生成完整 Spec。同一个对话继续说：

```
拆分方案我接受。现在为 redlite-auth 生成完整 Spec 文档。

结构要求：
1. 模块边界（包含什么 / 不包含什么）
2. 核心场景（用 WHEN/THEN/AND，覆盖正常 + 异常路径）
3. 数据结构（请求/响应字段，含类型与约束）
4. 验收标准（ACCEPTANCE，可勾选）

技术约束：
- 后端 JWT 鉴权，Token 含 7 天过期
- 密码加密存储，不明文返回
- 输出 Markdown，保存为 specs/redlite-auth.md
```

AI 产出的 Spec 关键部分长这样：

```markdown
# redlite-auth Spec

## 模块边界
包含：注册、登录、JWT 签发与校验
不包含：第三方登录、找回密码（后续单独 Spec）

## 核心场景
WHEN 用户用未注册邮箱注册 → THEN 创建账号 + 返回 JWT
WHEN 邮箱已存在 → THEN 返回 409 "邮箱已注册"
WHEN 登录密码错误 → THEN 返回 401，不暴露"是邮箱还是密码"错
WHEN 携带过期 Token 访问 → THEN 返回 401 要求重新登录

## 验收标准 ACCEPTANCE
[ ] 注册成功返回的响应体不含明文密码
[ ] Token 过期后受保护接口返回 401
[ ] 错误提示不泄露账号是否存在
```

**这就是 AI 能精准执行的 Spec——边界清晰、场景具体（WHEN/THEN）、验收可勾选。** 剩下 4 个单元用同一个模板改名字和字段即可。

## 速查清单：拆 Spec 自检表

每拆一个单元，对着勾一遍：

- [ ] 这个单元能**单独跑验收**吗？
- [ ] 它的数据来自**同一个源**吗？
- [ ] 做完**页面上看得到效果**吗？
- [ ] 有没有"被多处复用"的部分该抽成独立单元（如卡片）？
- [ ] Spec 里写了**异常路径**（不只是正常路径）吗？
- [ ] 验收标准**可勾选、可量化**吗？

三条标准全过 = 粒度对了；过不了就再拆或合并。

## 小结

- 拿到需求第一步是**拆 Spec**，不是让 AI 一把梭。
- 拆分三标准：**独立可测试、数据边界清、可独立交付**。
- 拆的时候把**标准 + 输出格式**一起喂给 AI，它才不会乱来。
- **粒度就是效率——拆对了，每个单元 AI 转化率 90%+。**
