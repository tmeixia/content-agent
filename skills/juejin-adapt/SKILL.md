# 掘金适配 Skill

把知识原子封装成一篇适合掘金（juejin.cn）的技术文章。

## 输入
- 单个知识原子，或几个相关原子组成的"实战合集"
- 主题大纲 `_outline.md`（可选，做系列文时定位）

## 输出
- 写到 `output/juejin/<topic>/<id>.md`
- 顶部给出标题候选 + 标签（tag）+ 分类（category）建议

## 平台调性
- 读者：一线开发者，懂技术、要"干货 + 能直接用"，对水文和标题党容忍度低于公众号。
- 语气：专业但有社区感，可以有"我"的实践经验、踩坑复盘；比知乎更"技术硬"，比极客更"轻快可读"。
- 长度：1500–3500 字，标题层级清晰，代码块密集。
- 结构惯例：前言（为什么写/能解决什么） → 正文（分小节，多代码多图） → 总结（要点回顾 + 一句互动）。
- 流量：靠首页推荐 + 站内搜索，**标签和标题关键词很重要**（调 `seo-title`）。
- 引流：可自然导流掘金小册 / 极客专栏 / 公众号，社区不反感技术向导流，但别硬广。

## 高质量文章的核心原则
- **图文并茂**：每篇至少 1 张图（流程图/架构图/对比图），用 `【图：<类型>：<核心信息描述>】` 占位
- **技术溯源**：讲"为什么有这个技术"，不只是"怎么用"——追溯设计动机和解决的问题
- **原理深度**：穿透 API 表层，走到机制层/设计取舍层，给出关键代码路径或执行流程
- **独到见解**：有自己立场（选型推荐/踩坑复盘/反常识结论），不搬运官方文档
- **拒绝泛泛**：每个结论必须有具体锚点（代码/场景/数据），禁止"一般来说"式的水词

## 步骤
1. 选切入点：掘金读者吃"我在项目里遇到 X，研究后发现 Y"的实战复盘式开头。
2. 标题：技术关键词 + 价值点，可用「｜」分隔。例：「React 性能优化｜为什么你的 React.memo 没生效」。调 `seo-title`。
3. 正文骨架：
   - **前言**：抛出真实场景/痛点，说明本文解决什么、读完能获得什么。
   - **正文小节**：每个小节一个知识点，标题用 `##`；先现象 → 再原理 → 再代码（反例/正例对照）→ 小结一句。
   - 代码完整可运行、关键行注释；鼓励配图/流程图位 `【配图：xxx】`。
   - 适当加"避坑提示""最佳实践"小贴士块（掘金读者爱收藏）。
   - **总结**：要点清单 + 一句互动引导（"你踩过哪些坑，评论区聊聊"）。
4. 给**标签建议**（取知识原子 tags + 高频搜索词，如 React、性能优化、前端）。
5. 术语对照 `knowledge/_glossary.md`。

## 与其他平台的边界（避免重复感）
- vs 知乎：掘金更技术硬核、代码更密、少故事多干货；知乎更代入式、重观点人设。
- vs 极客：掘金免费、轻快、单篇独立；极客付费、成体系、连载可互引。
- vs 公众号：掘金读者读长文、读代码；公众号要短、要钩子、代码要截断。

## 风格约束（沉淀区——人审后把经验写这里）
- 标题要有技术关键词（利于站内搜索），但禁止震惊体。
- 代码必须可运行（用 `code-example-gen` 校验），掘金读者会直接复制跑。
- 结尾留一句真实的互动引导，提升评论/点赞（影响推荐权重）。
- 标签建议精准，3–5 个，别堆无关词。
- 提示块用 `> **避坑提示**：` / `> **反直觉**：` 这类**纯文本加粗**，不要用 emoji（掘金虽常见 emoji，但本项目默认不写 emoji，除非人工要求）。
- 禁止"一般来说""通常情况下""众所周知"等水词，每个结论必须有具体锚点。
- 原理深度必须走到"设计取舍层"，不能停在"API用法层"。
