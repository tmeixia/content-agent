# 掘金小册 Skill

把一个主题，产出成体系的掘金小册（一篇一文，贯穿一个最小实战项目）。

**核心写作原则（贯穿全文，必须落地）：**
- **图文并茂**：每篇至少 2 张图（架构图 / 流程图 / 对比图 / 时序图），用 `【图：xxx】` 占位并描述图应表达的核心信息
- **技术溯源**：讲"为什么有这个技术"，而不只是"怎么用"——追溯设计动机、历史演进、解决的问题
- **原理深度解析**：穿透 API 表层，进入源码级 / 机制级理解，给出关键代码路径或执行流程
- **独到见解**：有自己的判断（选型对比、踩坑复盘、反常识结论），不搬运官方文档，不写"大家都知道"式的废话
- **拒绝泛泛而谈**：每个结论必须配具体代码、具体场景或具体数据，禁止"一般来说""通常情况下"式的水词

## 输入
- 小册大纲 `output/juejin/booklet-<topic>/00-booklet-outline.md`（篇目索引 + 项目说明）
- 本篇在大纲中的编号与标题
- 课程/项目事实约定（如本主题：`knowledge/sdd-harness-fullstack/_course-conventions.md`）

## 输出
- 写到 `output/juejin/booklet-<topic>/<NN>-<slug>.md`
- 顶部 frontmatter：小册名、本篇序号/标题、前后篇、demo 项目

## 定位（三者别混）
| | 慕课视频课 | 掘金单篇(juejin-adapt) | 掘金小册(本 skill) |
|---|---|---|---|
| 媒介 | 视频口播录屏 | 单篇图文 | 成体系图文（多篇） |
| 受众 | 可零基础、手把手 | 泛开发者 | **有基础、能自学** |
| 关系 | 节与节连贯 | 篇与篇独立 | **篇与篇连贯、贯穿一个项目** |
| 价值 | 带练 | 单点干货 | **方法论 + 可复用工具书** |

## 受众与口吻
- 受众：有一定基础、想把技术用好的进阶开发者。**不做零基础铺垫、不写"你看这里"视频话术。**
- 口吻：**结论先行**，干货密度高，像同行间分享而非课堂。可以有"我"的实战经验和踩坑复盘。
- 篇幅：单篇读 10–15min。

## 优质小册的 6 个核心特征（写作时逐条自检）

### 特征 1：图文并茂，视觉密度高
- **每篇至少 2 张图**，类型包括：
  - 🏗️ **架构图**：系统/模块的整体关系，用于开篇定位
  - 🔄 **流程图**：执行链路、数据流向，用于原理讲解
  - ⚖️ **对比图/表格**：方案A vs 方案B，用于选型决策
  - 📊 **时序图**：多组件交互顺序，用于异步/并发场景
  - 🗺️ **思维导图**：知识体系定位，用于篇首/篇尾总结
- 图片占位格式：`【图：<类型>：<一句话描述图应表达的核心信息>】`
- **反面示例**：`【图：流程图】`（太空）
- **正面示例**：`【图：流程图：SDD五步闭环——需求→Spec→Harness→AI生成→人工Review，标注每步的输入输出和决策点】`

### 特征 2：技术溯源，讲"为什么"
- **每个核心技术点，先回答三个问题**：
  1. 这个技术/模式是为了解决什么问题而被创造出来的？
  2. 在它之前，大家是怎么做的？那种做法有什么痛点？
  3. 它的核心设计取舍是什么？（不是银弹，有什么代价？）
- **不要**："React Hooks 是 React 16.8 引入的新特性，它让你在不写 class 的情况下使用 state……"（搬运文档）
- **要**："在 Hooks 之前，逻辑复用靠 HOC 和 Render Props，但这两种模式都会导致'嵌套地狱'——组件树每复用一个逻辑就多一层嵌套。Hooks 的本质是把'组件逻辑'从'组件结构'中解耦出来……"

### 特征 3：原理深度，穿透 API 表面
- **三层递进结构**（每个核心知识点都要走到第三层）：
  - **L1 用法层**：API 怎么调、参数是什么（快速带过，读者能查文档）
  - **L2 机制层**：底层怎么运行的、执行顺序是什么、边界条件是什么
  - **L3 设计层**：为什么这样设计、有什么替代方案、设计者当时的取舍
- **源码引用**：给出关键代码路径或伪代码，不要求贴完整源码，但要指出"关键在哪几行"
- **执行流程可视化**：用编号步骤 + 流程图展示执行链路，不能只用文字描述

### 特征 4：独到见解，有自己的判断
- **选型对比要有立场**：不只列优缺点，要给出"在X场景下我推荐Y，因为Z"
- **踩坑复盘要具体**：不说"要注意性能问题"，要说"我们在X场景下遇到了Yms的延迟，根因是Z，解决方式是…"
- **反常识结论要高亮**：如果结论违反直觉，用 `> **反直觉**：` 块单独标出，并给出论证
- **技术判断框架**：给读者一个可以迁移的判断框架，而不只是"这个好/那个差"

### 特征 5：拒绝泛泛，每个结论都有锚点
- **禁止**以下水词模式：
  - ❌ "一般来说……" → ✅ 给出具体场景/案例
  - ❌ "通常情况下……" → ✅ 给出具体条件/阈值
  - ❌ "众所周知……" → ✅ 直接讲，或给出反例挑战"常识"
  - ❌ "性能更好" → ✅ 给出数据/复杂度/对比测试结论
  - ❌ "建议在实际项目中灵活使用" → ✅ 给出你的实际用法
- **代码必须可运行**：掘金读者直接复制跑，不能是伪代码片段（除非明确标注为伪代码）

### 特征 6：可抄走，收藏价值高
- **每篇至少有 1 个"可直接抄走"的硬货**：
  - 完整可运行的代码片段（含上下文，不依赖外部未知变量）
  - 完整的 Prompt（含拆分标准/输出格式/技术约束）
  - 可直接使用的配置文件 / 工具链配置
- **每篇至少有 1 个速查清单或对照表**（决定收藏率）

## 单篇体例（7 段式，必须齐全）

1. **痛点开篇**（~200字）
   - 一句话戳中读者真实困惑 / 翻车场景，直接抛结论（不铺垫）
   - 给一个真实场景的代码反例（"你可能也这么写过，然后在X情况下翻车了"）

2. **技术溯源**（~300字）
   - 这个技术/模式为什么存在？解决什么问题？
   - 之前怎么做？痛点在哪？
   - 配 `【图：演进对比图 / 问题场景图】`

3. **原理深度解析**（~600字，本篇最硬核的部分）
   - 三层递进：用法 → 机制 → 设计取舍
   - 配 `【图：执行流程图 / 架构图】`
   - 关键源码路径或伪代码（标注"关键在这几行"）
   - 用编号步骤拆解执行链路

4. **实战落地**（~500字）
   - 完整可运行代码（含注释，关键行标注）
   - 反例 vs 正例对照（标注每处差异的原因）
   - 配 `【图：效果对比图 / 结果截图】`（如适用）

5. **可抄走的硬货**（~300字）
   - **完整 Prompt**（含拆分标准/输出格式/技术约束，不要"让AI做一下"这种残缺Prompt）
   - 或：完整配置文件 / 工具链配置 / 可复用代码模板
   - 这是小册区别于视频课的核心价值——读者要能直接抄走

6. **速查清单 / 表格**（~200字）
   - 可勾选 checklist 或对照表（小册读者爱收藏，决定收藏率）
   - 格式：Markdown 表格或 `- [ ]` 列表

7. **独到小结 + 动手点**（~150字）
   - 1–2 句加粗结论（要有自己的判断，不是复述前文）
   - 一个 5 分钟可做的小任务

> 允许 ✅❌⚠️ 作为功能标记（沿用课程既有风格，不算装饰 emoji）。

## 小册级一致性约定
- **贯穿同一个 demo 项目**（如本主题：简版小红书 RedLite，仅 登录注册/发帖/列表/详情）。
- **前后篇钩子**：开头一句"上一篇我们…"、结尾一句"下一篇…"，让篇与篇连起来。
- **术语/事实统一**：技术栈、指令（opsx）、Spec 格式、产品代号，全部对照事实约定文件，跨篇不得漂移。
- **商标规避**：涉及真实品牌（如"小红书"）用代号 + "类/简版图文社区"描述。

## 与视频讲师稿的复用
- 方法论内核（SDD/Harness/opsx/五步闭环/金句）复用，但**改写为图文、精炼、去手把手**。
- 视频稿的"口播+演示分镜" → 小册的"结论 + 完整Prompt + 关键代码 + 清单"。

## 风格约束（沉淀区——人审后把经验写这里）
- 严禁把视频讲师稿直接搬来删——要按图文逻辑重组：结论先行、硬货可抄、清单可收藏。
- 每篇至少 2 张图、1 个"可抄走的完整 Prompt/代码"和 1 个"速查清单/表格"，否则不算合格小册篇。
- 不堆零基础铺垫；读者卡基础的地方给一句话链接到对应基础篇，不展开。
- 禁止"一般来说""通常情况下""众所周知"等水词，每个结论必须有具体锚点。
- 原理深度必须走到"设计取舍层"，不能停在"API用法层"。
- 提示块用 `> **避坑提示**：` / `> **反直觉**：` 这类纯文本加粗，不要用 emoji。
